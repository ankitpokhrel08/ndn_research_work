\begin{abstract}
This report documents the complete experimental setup and analysis results for an unsupervised anomaly detection system targeting two attack classes in Named Data Networking (NDN): Interest Flooding Attack (IFA) and Content Poisoning (CP). The study uses a dual-platform approach to strengthen result validity. \textbf{Part~I} describes ndnSIM~2.x (ns-3.35) simulations across three topologies --- Tree-12, DFN-12, and Dumbbell-10 --- under IFA and CP attack conditions, evaluated using a dual per-node statistical detector. \textbf{Part~II} describes miniNDN emulation experiments on the same three topologies running real NFD (Named Data Networking Forwarding Daemon) processes, evaluated using an Isolation Forest machine learning model for IFA detection.

A core finding motivating the dual-detector design in Part~I is that IFA and CP produce directionally opposite changes in the same NDN traffic features. In Part~I, the dual z-score detector achieves pre-attack scores of 1.4--1.8 and post-attack lifts of 40--86 points across all topology-attack combinations, with detection latencies of 1--2 seconds for IFA and 1--9 seconds for CP. In Part~II, the Isolation Forest achieves 91.2--98.7\% attacker detection at the operational threshold across all three topologies with false positive rates below 4.3\% on clean normal traffic. Both platforms independently confirm that a globally trained model generalises across all three topologies without retraining, validating the topology-invariance of NDN attack signatures.
\end{abstract}

% ============================================================
\section{Introduction}

Named Data Networking (NDN) replaces IP's host-centric addressing with content-centric forwarding: packets carry named content rather than source/destination addresses, and routers cache responses in a Content Store (CS) for reuse. The Pending Interest Table (PIT) tracks outstanding requests, and the Forwarding Information Base (FIB) routes interests toward producers.

Two attacks exploit the stateless nature of NDN forwarding:

\begin{itemize}[noitemsep]
  \item \textbf{Interest Flooding Attack (IFA):} A compromised consumer floods the network with interests for non-existent or unresolvable content. Fake interests propagate to producers and occupy PIT entries for their full lifetime, causing saturation that blocks legitimate forwarding.
  \item \textbf{Content Poisoning (CP):} An attacker injects interests for a large number of unique, rarely-requested names. Because the content is served by legitimate producers, it enters router Content Stores through normal NDN caching, continuously evicting popular content. Legitimate consumers suffer higher cache miss rates and increased producer load.
\end{itemize}

This work validates anomaly detection for these attacks on two complementary platforms. ndnSIM provides a controlled simulation environment for evaluating both IFA and CP simultaneously with deterministic traffic and ground-truth timing. miniNDN provides an emulation environment where real NFD processes handle actual NDN packet forwarding, capturing system-level behaviours (OS scheduling, NFD implementation quirks, realistic timing jitter) that simulation cannot model. The agreement of key findings across both platforms strengthens confidence in the results.

\section{Simulation Platform}

\subsection{Environment}

All simulations run inside a Docker container based on Ubuntu~20.04 using:

\begin{itemize}[noitemsep]
  \item \textbf{ns-3.35} --- discrete-event network simulator
  \item \textbf{ndnSIM~2.x} \cite{ndnsim-repo} --- NDN protocol module for ns-3, providing NFD-based forwarding, PIT/FIB/CS tables, and consumer/producer application helpers
  \item \textbf{C++17} compiler toolchain
\end{itemize}

Scenarios reside in a \texttt{ndn-research/} directory compiled via Waf. Custom applications (e.g., the non-responding producer for IFA) reside in \texttt{extensions/} and are compiled into the scenario library automatically.

\subsection{Tracers and Output Format}

Each simulation produces four output files per run:

\begin{enumerate}[noitemsep]
  \item \textbf{L3RateTracer} (\texttt{*-rate-trace.txt}) --- per-face, per-node, per-second counters for InInterests, OutInterests, InData, OutData, InNacks, OutNacks, InSatisfiedInterests, InTimedOutInterests.
  \item \textbf{CsTracer} (\texttt{*-cs-trace.txt}) --- per-node, per-second Content Store hit and miss counts.
  \item \textbf{AppDelayTracer} (\texttt{*-app-delays.txt}) --- per-packet application-layer delay and retransmission counts.
  \item \textbf{Ground truth} (\texttt{*-ground-truth.csv}) --- per-second labels: \texttt{normal}, \texttt{ifa}, or \texttt{cp}.
\end{enumerate}

\section{Topologies (Shared Across Both Platforms)}

The same three topologies are used in both ndnSIM (Part~I) and miniNDN (Part~II), enabling direct cross-platform comparison. All point-to-point links operate at 10\,Mbps with 10\,ms propagation delay unless noted. Content Store capacity is 100 entries per node.

\subsection{Tree-12}

Hierarchical topology: two producers (\texttt{p1}, \texttt{p2}) connect to root router \texttt{r1}, which branches through \texttt{r2}, \texttt{r3}, \texttt{r4} to six consumers (\texttt{c1}--\texttt{c6}). All consumer traffic passes through backbone routers \texttt{r1}--\texttt{r2}, making this topology susceptible to network-wide flooding.

\subsection{DFN-12}

Identical node set to Tree-12 with two additional cross-links (\texttt{r1}--\texttt{r4} at 15\,ms and \texttt{r2}--\texttt{r3} at 15\,ms) creating a denser mesh with redundant forwarding paths.

\subsection{Dumbbell-10}

Bottleneck topology with three consumers (\texttt{c1}--\texttt{c3}) and two producers (\texttt{p1}, \texttt{p2}) connected through a single bottleneck node:

\begin{lstlisting}[frame=none,backgroundcolor=\color{white}]
c1, c2, c3 -> r1 -> r2 -> bottleneck -> r3 -> r4 -> p1, p2
\end{lstlisting}

All traffic concentrates at the bottleneck, producing the strongest detection signals of the three topologies.

\section{ndnSIM Traffic Scenarios}

Nine simulation runs (3 topologies $\times$ 3 scenarios) each last 600 seconds. Attack traffic starts at $t = 300$\,s, giving balanced 300\,s normal and 300\,s attack segments.

\subsection{Normal Traffic}

All consumers use \texttt{ConsumerZipfMandelbrot} at 10 interests/s with catalog $N = 100$, $q = 0.0$, $s = 0.7$. Both producers serve \texttt{/ndn} with 1024-byte payloads. Routes are computed via \texttt{GlobalRoutingHelper}.

\subsection{Interest Flooding Attack (IFA)}

Legitimate consumers use \texttt{ConsumerCbr} at 10 interests/s (flat distribution, not Zipf-Mandelbrot, to keep the pre-attack baseline fixed). At $t = 300$\,s, consumer \texttt{c1} launches four concurrent attack streams:

\begin{table}[H]
\centering
\caption{IFA attack stream configuration (all from \texttt{c1}, starting at $t = 300$\,s)}
\begin{tabular}{lcc}
\toprule
\textbf{Prefix} & \textbf{Rate (int/s)} & \textbf{InterestLifetime} \\
\midrule
\texttt{/evil/fake/video}       & 40 & 2000\,ms \\
\texttt{/evil/random-attack}    & 30 & 1500\,ms \\
\texttt{/evil/fake/image}       & 20 & 1000\,ms \\
\texttt{/evil/random-attack-2}  & 10 & 4000\,ms \\
\midrule
\textbf{Total}                  & \textbf{100} & --- \\
\bottomrule
\end{tabular}
\end{table}

A custom \texttt{BlackholeProducer} is installed for the \texttt{/evil} prefix. It accepts interests but never sends Data, causing PIT entries to hold until timeout. Without this, interests would either be satisfied (no PIT pressure) or NACKed immediately (no timeout accumulation).

\subsection{Content Poisoning (CP)}

Legitimate consumers use Zipf-Mandelbrot as in the normal scenario. At $t = 300$\,s, consumers \texttt{c1} and \texttt{c6} (or \texttt{c1} and \texttt{c3} in Dumbbell) each launch 50 \texttt{ConsumerCbr} applications requesting unique junk names under \texttt{/ndn/junk/} at 2 interests/s each (200 total junk interests/s). Because junk names fall under \texttt{/ndn}, producers serve them legitimately, filling the CS and continuously evicting popular cached content.

\section{ndnSIM Detection Methodology}

\subsection{Motivation: Opposite Attack Signatures}

IFA and CP produce directionally opposite changes in the same features, making a single detector insufficient:

\begin{table}[H]
\centering
\caption{Feature response directions under IFA vs.\ CP}
\begin{tabular}{lcc}
\toprule
\textbf{Feature} & \textbf{Under IFA} & \textbf{Under CP} \\
\midrule
Satisfaction Ratio & $\downarrow$ collapses to $\approx 0$      & $\uparrow$ stays high (junk is satisfied) \\
Timeout Ratio      & $\uparrow$ spikes sharply                  & $\approx$ unchanged \\
Cache Hit Ratio    & $\approx 0$ (no hits on \texttt{/evil})    & $\downarrow$ (legitimate content evicted) \\
InInterests        & $\uparrow$ +100 fake interests/s           & $\uparrow$ moderate (+200 junk/s) \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Dual Per-Node Static Baseline Detector}

For each node, a baseline mean $\mu_f$ and standard deviation $\sigma_f$ is computed from the first 200 seconds of the pure normal scenario. No attack data is used.

\paragraph{IFA Sub-Detector.}
Directional z-scores on three features ($F_{\text{IFA}}$: satisfaction\_ratio$\downarrow$, timeout\_ratio$\uparrow$, InInterests$\uparrow$). A timestep is flagged if the mean directional z-score exceeds 2.5. The IFA score is the fraction of flagged steps in a 3-step rolling window, scaled to 0--100.

\paragraph{CP Sub-Detector.}
Directional z-scores on two features ($F_{\text{CP}}$: cache\_hit\_ratio$\downarrow$, InInterests$\uparrow$). A timestep is flagged if the mean directional z-score exceeds 2.0. The CP score is the fraction of flagged steps in a 10-step rolling window, scaled to 0--100. The wider window smooths the oscillating burst pattern of CP traffic.

\paragraph{Combined Score and Alarm.}
$\text{anomaly\_score} = \max(\text{IFA\_score},\ \text{CP\_score})$. An alarm is raised when anomaly\_score~$> 50$.

\section{Emulation Platform}

\subsection{Environment}

miniNDN~\cite{minindn} is a Python-based network emulator built on Mininet that provisions real Linux network namespaces and runs actual NFD (Named Data Networking Forwarding Daemon) processes on each emulated node. Unlike ndnSIM, which implements a simplified NDN model in C++, miniNDN executes the same NFD binary used in production NDN deployments. This captures system-level effects such as OS process scheduling overhead, NFD's internal buffering behaviour, and realistic timing jitter --- all absent from simulation.

The experimental stack consists of:
\begin{itemize}[noitemsep]
  \item \textbf{miniNDN} --- topology provisioning and experiment orchestration
  \item \textbf{NFD (Named Data Networking Forwarding Daemon)} --- actual NDN forwarding process per node
  \item \textbf{NLSR (Named-data Link State Routing)} --- distributed routing protocol for FIB computation
  \item \textbf{Custom Python logging scripts} --- poll per-node NFD management counters every $\sim$10 seconds and write records to JSONL files
\end{itemize}

\subsection{Data Collection}

Each node's NFD management interface exposes cumulative counters (nPitEntries, nInInterests, nOutInterests, nInData, nInNacks, nOutNacks, nSatisfiedInterests, nUnsatisfiedInterests, nCsEntries, nHits, nMisses) that are polled at approximately 10-second intervals and written to per-node JSONL log files. The polling interval is inherently less precise than ndnSIM's deterministic 1-second output due to OS scheduling, resulting in variable inter-sample intervals that are accounted for during feature extraction.

A 10-minute (600-second) warmup period is discarded per node to eliminate cache cold-start artefacts before features are computed.

\subsection{Attack Scenario}

The miniNDN experiments cover \textbf{IFA only}. In each topology, node \texttt{c1} generates a stream of interests for non-existent content names at a high rate starting from experiment onset (no warmup split --- the entire log is labelled attack after the warmup removal). The attack exploits the same PIT-exhaustion mechanism as in ndnSIM: interests for unresolvable names accumulate in router PITs until their lifetime expires, competing with legitimate traffic.

CP data was not collected in the miniNDN campaign. This is acknowledged as a limitation; the ndnSIM results (Part~I) provide CP coverage.

\section{miniNDN Model: Isolation Forest}

\subsection{Algorithm and Configuration}

An Isolation Forest \cite{iforest} is trained on the clean normal traffic data only (no attack labels used). The model is wrapped in a scikit-learn Pipeline with a StandardScaler preprocessing step.

\begin{table}[H]
\centering
\caption{Isolation Forest hyperparameters}
\begin{tabular}{ll}
\toprule
\textbf{Parameter} & \textbf{Value} \\
\midrule
Number of trees          & 200 \\
Contamination            & 0.01 (1\% assumed anomaly rate in training data) \\
Max samples              & auto \\
Random state             & 42 \\
Preprocessing            & StandardScaler \\
Anomaly threshold $\theta$ & 0.0 (raw \texttt{decision\_function} score) \\
Training samples         & 126,423 (combined normal traffic, all 3 topologies) \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Score Normalisation}

Raw \texttt{decision\_function} scores are mapped to a 0--100\% scale for interpretability:

\begin{equation}
\text{score}_{[0,100]} =
\begin{cases}
30 + \dfrac{s - \theta}{s_{\max} - \theta} \times 70 & s \geq \theta \quad \text{(normal side)} \\[8pt]
30 \times \dfrac{s - s_{\min}}{\theta - s_{\min}} & s < \theta \quad \text{(anomaly side)}
\end{cases}
\end{equation}

where $s$ is the raw score, $\theta = 0.0$ is the anomaly boundary, and $s_{\min}$, $s_{\max}$ are the minimum and maximum training scores. A score below 30\% corresponds to the strict detection threshold; below 50\% corresponds to the operational threshold.

\subsection{Tree Topology: Network-Wide IFA Effect}
\label{sec:tree-finding}

The elevated benign consumer detection rates in the Tree topology (51--58\%) are not a false alarm failure --- they are a real network-wide effect. The tree's hierarchical structure forces all consumer traffic through backbone routers \texttt{r1} and \texttt{r2}. When \texttt{c1} floods these routers with IFA traffic, PIT exhaustion and NACK propagation degrade the forwarding service for \emph{all} consumers on all branches. Every benign consumer experiences anomalous satisfaction\_ratio, nack\_rate, and interest rates even though only \texttt{c1} is generating attack traffic.

In DFN and Dumbbell topologies, alternative forwarding paths and physical separation of consumer branches prevent the flood from reaching all nodes, keeping benign consumer detection rates below 7.3\%.

\subsection{Global vs.\ Per-Topology Model}

A single globally trained Isolation Forest (trained on all three topologies' normal data combined) achieves attacker detection rates statistically indistinguishable from three separate topology-specific models. The global model is preferred in practice as it requires no topology-specific calibration and generalises to deployments where topology may not be known in advance.

\section{Key Findings}
\label{sec:findings}

\begin{enumerate}

  \item \textbf{IFA and CP produce opposite feature signatures (ndnSIM).} IFA causes satisfaction\_ratio to collapse and timeout\_ratio to spike; CP causes satisfaction\_ratio to stay elevated while cache\_hit\_ratio drops. A single-direction detector cannot cover both attack types simultaneously, motivating the dual sub-detector architecture.

  \item \textbf{Attack signatures are topology-invariant (both platforms).} The global model performs identically to topology-specific models on every evaluation scenario in both ndnSIM and miniNDN. NDN attack signatures in the chosen feature space generalise across hierarchical, mesh, and bottleneck network structures without retraining. This is the single most practically important finding: a detector deployed at one site transfers to any other NDN topology.

  \item \textbf{Sub-second to low single-digit detection latency (ndnSIM).} IFA detected within 1--2 seconds; CP within 1--9 seconds. Both are within the NDN Interest lifetime window (1--4 seconds), meaning alarms are raised before significant PIT or cache damage accumulates.

  \item \textbf{Spatial attack fingerprint via router detection pattern (both platforms).} Routers on the attack forwarding path alarm at high rates; routers on unaffected branches remain near zero. This allows approximate attacker localisation without explicit routing table access.

  \item \textbf{IFA propagates to producers (miniNDN).} Producer \texttt{p1}, on \texttt{c1}'s primary forwarding path, is detected at 90.5\%, 72.4\%, and 46.5\% across the three topologies. IFA floods Interests all the way to producer nodes, not just intermediate routers --- a finding not easily observable in simulation where producer-side behaviour is simplified.

  \item \textbf{Tree topology exhibits network-wide IFA flooding (miniNDN).} Benign consumers flag at 51--58\% in the tree topology, versus $<7.3\%$ in mesh and dumbbell. The hierarchical structure makes IFA a network-wide QoS degradation event affecting all branches, not just the attacker's branch.

  \item \textbf{Cross-platform agreement strengthens validity.} The Isolation Forest (miniNDN, ML-based) and the dual z-score detector (ndnSIM, statistical) independently confirm topology-invariant detection and spatial attack fingerprinting using different methods, different simulators, and different feature sets. Agreement on the core findings reduces the risk that either result is an artefact of its platform.

\end{enumerate}

% ============================================================
\section{Limitations}
\label{sec:limitations}

\begin{enumerate}[noitemsep]

  \item \textbf{CP not evaluated in miniNDN.} Cache Poisoning data was not collected in the emulation campaign. The CP results (Part~I) rely entirely on ndnSIM.

  \item \textbf{Victim consumers not directly detected.} In ndnSIM, benign consumers affected by IFA score near zero because their own traffic generation metrics appear normal (they continue sending interests at their configured rate). In miniNDN, the tree topology does flag benign consumers, but this is due to real network-wide QoS degradation, not victim-specific anomaly detection.

  \item \textbf{Simulated environment for ndnSIM.} ndnSIM models idealised 10\,Mbps point-to-point links with fixed delays. Real deployments exhibit variable link quality, asymmetric paths, and dynamic topology changes.

  \item \textbf{miniNDN polling interval (10\,s) vs.\ ndnSIM (1\,s).} The coarser polling interval in miniNDN smooths fast transients (e.g., PIT growth rate becomes non-discriminating at $D = 0.028$). Sub-second polling would likely improve PIT-based feature utility.

  \item \textbf{Single attack per run.} Concurrent mixed attacks (simultaneous IFA and CP from different nodes) were not evaluated on either platform.

  \item \textbf{Detector threshold sensitivity not evaluated.} Z-score thresholds (2.5 for IFA, 2.0 for CP) and Isolation Forest contamination (0.01) were set based on observed training distributions. Systematic sensitivity analysis was not performed.

\end{enumerate}

% ============================================================
\section{Design Decisions and Debugging}

Several issues were identified and resolved during development:

\begin{enumerate}[noitemsep]

  \item \textbf{(ndnSIM) Routable attack prefixes.} Early IFA scenarios used prefixes under \texttt{/ndn}, which the standard Producer satisfied. Switching to \texttt{/evil} with a BlackholeProducer ensured genuine PIT timeouts.

  \item \textbf{(ndnSIM) Cache-insensitive baseline.} The initial CP scenario used sequential \texttt{ConsumerCbr} with zero cache re-requests, making cache pollution unmeasurable. Replacing with \texttt{ConsumerZipfMandelbrot} established a high-cache-hit baseline.

  \item \textbf{(ndnSIM) CsTracer reporting zeros.} The \texttt{CsTracer} originally reported all-zero values due to a missing full header inclusion for \texttt{nfd::Forwarder}, preventing connection to \texttt{afterCsHit}/\texttt{afterCsMiss} signals.

  \item \textbf{(ndnSIM) NFD strategy segfault.} The patched ndnSIM's \texttt{Strategy::sendData()} omitted a return statement in the no-PitToken code path, causing a segfault on CS hits. Fixed by adding \texttt{return true;}.

  \item \textbf{(miniNDN) Delta-based feature computation.} An earlier preprocessing version used cumulative totals for cache\_hit\_ratio and satisfaction\_ratio, causing these features to drift over time as denominators grew. Switching to per-interval deltas stabilised the features across all time windows.

\end{enumerate}
